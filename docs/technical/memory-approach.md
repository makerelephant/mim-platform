## The Core Challenge

Claude is stateless. Every API call starts with a blank slate. It has no memory of the email it classified yesterday, the correction the CEO made last week, or the document uploaded three months ago.

This is the fundamental tension at the centre of building an intelligent business platform on top of a stateless AI: the platform must supply the memory the model lacks.

In Motion solves this through three distinct memory layers, each operating at a different timescale and serving a different purpose. Together, they are what transforms a general-purpose language model into a system that knows your business.

## The Three Layers

```
┌─────────────────────────────────────────────────────┐
│  LAYER 3: BEHAVIORAL ADAPTATION (Learning Memory)   │
│  Brain changes how it operates based on patterns.   │
│  Timescale: weeks to months                         │
│  Where: brain.behavioral_rules, brain.decision_log  │
└──────────────────────────┬──────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────┐
│  LAYER 2: DERIVED INTELLIGENCE (Synthesis Memory)   │
│  Brain generates new knowledge from aggregate data. │
│  Timescale: days to weeks                           │
│  Where: brain.derived_insights, brain.instructions  │
└──────────────────────────┬──────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────┐
│  LAYER 1: ACCUMULATED KNOWLEDGE (Storage Memory)    │
│  Raw facts stored and retrievable via RAG.          │
│  Timescale: immediate, persistent                   │
│  Where: knowledge_base, knowledge_chunks, pgvector  │
└─────────────────────────────────────────────────────┘
```

## Layer 1: Accumulated Knowledge (Storage Memory)

### What it is

Raw facts stored in the entity graph and knowledge base. Every email, Slack message, uploaded document, and ingested piece of data. More data = more to retrieve.

### How it works

Everything ingested goes through the single ingestion point. Text is extracted, chunked (~500 tokens per chunk), and embedded via OpenAI `text-embedding-3-small` (1536 dimensions). Embeddings are stored in `brain.knowledge_chunks` (pgvector). Correspondence is embedded separately in `brain.correspondence_chunks`.

When the brain needs to answer a question, it embeds the question, finds the most semantically similar stored chunks via cosine similarity, and injects the relevant context into the Claude prompt.

### Limitations of Layer 1 alone

Storage is a filing cabinet, not intelligence. If you store 10,000 emails and ask "what patterns emerge in how our partnership deals progress?", RAG retrieval will find relevant emails but will not have synthesised the pattern. Layer 1 answers "what do we have on X?" It cannot answer "what does it mean?"

## Layer 2: Derived Intelligence (Synthesis Memory)

### What it is

New knowledge that didn't exist in any single input — patterns, correlations, predictions, and insights generated by analysing data in aggregate. This is the layer that makes the brain proactive rather than reactive.

### How it works

A synthesis agent runs on schedule (daily + after significant data batches). It reads recent activity, classifications, CEO actions, enrichment results, and entity feedback. It identifies patterns across entities, time periods, and categories. It writes new derived insight entries with natural language descriptions, supporting evidence, confidence levels, and applicable scope.

Derived insights are also embedded for RAG retrieval — so they surface automatically when the brain is asked a related question.

### Examples

- "Clubs affiliated with AYSO in the Southwest have 3× higher conversion rates than independent clubs."
- "Entities discovered through Instagram have higher engagement but lower purchasing volume."
- "When the CEO ignores newsletter-type tasks 80%+ of the time, stop creating them."
- "Partnership deal velocity correlates with coach involvement in the design process."

### Layer 2 and the Instruction Engine

CEO-authored instructions are a manual form of Layer 2 memory. When the CEO says "always flag emails from term sheets as critical" or "include the Q1 strategy in this week's report," they are creating explicit synthetic knowledge that the brain uses to modify its behaviour. Over time, the synthesis agent will generate these kinds of rules automatically from observed patterns.

## Layer 3: Behavioral Adaptation (Learning Memory)

### What it is

The system changes how it operates based on what it has learned. Not just "more data" — fundamentally different behaviour on future inputs. This is the layer that makes the brain get better with use.

### How it works

An adaptation agent runs weekly and after significant CEO override patterns. It analyses `brain.decision_log` for systematic patterns in corrections. When it finds a consistent pattern (e.g., CEO always upgrades priority on emails from `.edu` domains), it proposes a behavioral rule. High-confidence rules (large sample, clear pattern) are auto-applied. Lower-confidence rules surface to the CEO for approval via a feed card. All rules are reversible.

Rules are stored in `brain.behavioral_rules` and loaded into scanner prompts, classifier configs, and enrichment strategies on every execution.

### Examples

- "Elevate priority for emails from domains matching existing investors."
- "Skip enrichment via web scanner for entities with KCS > 0.8."
- "Scanner accuracy for partnership emails is 95% but for investor emails only 70% — add richer context for investor classification."

## The Compounding Effect

Each memory layer feeds the next:

**Month 1:** Brain processes 2,000 emails. Raw classification. CEO corrects ~30%. System writes 15 behavioral rules. KCS across entities: 0.15 avg.

**Month 3:** 15 rules in every Gopher prompt. CEO corrections drop to ~10%. 40+ derived entity patterns. Report generator includes insights CEO didn't request. KCS: 0.35 avg.

**Month 6:** System predicts which new organisations are high-value before human interaction. Gopher queues self-prioritise. Brain proactively flags churn risk. KCS: 0.55 avg.

**Month 12:** Brain knows the business better than any single employee. Perfect recall across every interaction, every correction, every pattern. Applies all context to every decision. KCS: 0.75+ avg.

Without these layers, every interaction is a cold start with a bigger database. With them, every interaction benefits from every previous interaction.

## How Claude Gets Memory at Runtime

Claude is stateless. What makes it appear to have memory is what we inject into its context window before each call. The pattern:

```
[ System Prompt ]
  → Harness rules (what the brain knows about this category)
  → Behavioral rules (what the brain has learned)
  → Active instructions (what the CEO has asked for)

[ Context Block ]
  → Entity dossier (everything known about the relevant entities)
  → RAG chunks (semantically relevant stored knowledge)
  → Recent activity (last N days of relevant correspondence)
  → Derived insights (patterns relevant to this query)

[ Input ]
  → The message / question / document to process
```

The richer and more accurate this context injection, the better Claude performs. This is why the three memory layers matter: they are the source material for context injection. A brain with 6 months of accumulated knowledge, derived insights, and learned behavioral rules provides dramatically better context than a brain that started yesterday.

## What This Is Not

**It is not fine-tuning.** We do not modify the Claude model weights. All memory is external — stored in Supabase, injected at runtime. This means memory is auditable, correctable, and does not require retraining.

**It is not a conversation history.** The clearing/canvas sessions persist conversation turns, but the deeper memory layers are not conversation-level — they are business-level. The brain remembers the business, not the chat.

**It is not a simple database.** Layer 1 is a database. But Layers 2 and 3 are synthetic — they generate knowledge that was never explicitly stored. The whole is greater than the sum of its parts.
